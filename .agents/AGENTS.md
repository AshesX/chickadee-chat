# AGENTS.md

Guidance for coding agents (the Gemini agent system and other AGENTS.md-aware tools) when working with code in this repository.

> **Doc sync:** this file mirrors [CLAUDE.md](../CLAUDE.md) at the repo root. When you change one, apply the same change to both — they are identical apart from this intro.

**Chickadee Chat** is a lightweight P2P desktop **voice / video / screen-share** app — "Discord Lite" — for small groups of **up to 8 users per room** (one unified "hybrid" room type: audio + optional golden-ratio video). Built to call friends and share a game screen. **Windows-first.** The media core, the high-contrast "Blaze" UI (Almost Black `#070D0D` + Blaze Orange `#FF6700`), and a portable Windows build are all complete; the renderer's large files have been decomposed and a `vitest` unit-test net guards the pure logic.

## Performance & code philosophy

Performance is a primary consideration in every change — the app's users are critical of any performance demands it imposes. Keep things DRY (Don't Repeat Yourself) and adhere to an ultra-lean, clean code philosophy.

## Current state

What the app does today:

- **Full-mesh P2P media:** mesh audio, video, and screen share (≤8 peers/room) with Windows system/game-audio capture (loopback). Configurable signaling URL + STUN/TURN, auto-reconnect + ICE restart.
- **Hybrid rooms (unified, 8-cap):** every room is one type — audio + optional video. Users flow between audio-only, camera, and screen-share without leaving. The full mesh is protected by an **asymmetric-quality model** (see below), not a low headcount. Spaces group rooms; Spaces are invite-code shareable, with optional per-Space signaling URL + join secret.
- **Stage spotlight ("golden ratio"):** at most **one** high-quality "stage" stream per room (a spotlighted screen *or* camera); everyone else's webcam is a compressed low-bitrate thumbnail. The stage is a server-arbitrated single slot (claim/release/take-over); screen shares auto-claim it, cameras spotlight manually. UI shape-shifts: **Voice Lounge** (0 videos) → **Gallery** (videos, none spotlighted) → **Theater** (one large stage tile + filmstrip). Viewing stays opt-in ("Watch"). Outbound bitrate is capped by an **upload budget** (Settings → Video) so a full room can't saturate a home uplink.
- **Voice controls:** input modes — Voice activation (VAD, with sensitivity threshold) / Push-to-Talk; global **PTT / mute / deafen** hotkeys (hold or toggle), captured system-wide via `uiohook-napi` + Electron `before-input-event`. Chromium noise **suppression**; mic volume + boost; live mic meter.
- **Output / awareness:** per-peer volume 0–200% (persisted by `userId`) + click-to-silence; "Normalize voices" listener-side auto-level; deafen; output-device selection; speaking indicators; manual status (Online / Idle / DND).
- **Identity:** customizable avatars (canvas crop → 128×128 WebP, synced space-wide), accent colors, display name.
- **Chat:** room chat + emoji reactions over the signaling relay; optional text-to-speech read-aloud with synced per-user voice categories.
- **Moderation (Space Owner + Room Moderator):** the Space Owner (gold crown; the reworked first-claim-wins `spaceOwners` record, now transferable) can kick a user from a room or the Space, **ban/unban** by stable `userId` (honor-system — evadable by identity reset, accepted), **Lock Space** (blocks newcomers; presence-roster members + owner still get in), and lock/unlock any active room. Each room also has a temporary **Room Moderator** (silver crown) — the longest-present member, derived server-side from the room Map's insertion order — who can kick-from-room + lock their room while the owner is away; the role passes on leave and resets when the room empties. Actions surface via right-click (USERS rows + participant tiles), the room/space context menus, and an owner-gated Moderation section in Space Settings (lock, transfer, ban list). All authority is enforced server-side (`canModerate`); bans + the space lock survive server restarts via an owner-client re-seed. **Room/space governance** rides the same ownership: space rename is owner-only, and the room list is server-validated — the owner manages every room, while a standard member may hold ONE self-created room (`Room.createdBy` stamp) that only they (or the owner) can rename/delete; legacy/unstamped rooms are owner-managed. Non-owners see "Leave Space" instead of "Delete Space" (same local-only removal).
- **P2P file transfer:** send file(s) to any **online Space member** (any room) from the sidebar USERS list — hover button (multi-select picker) or **drag-and-drop onto the row**. Receiver gets an accept prompt → native Save As → the file streams over a **dedicated WebRTC DataChannel per FILE** (independent of the media mesh), written to disk via main-process IPC (`.part` → rename on completion). **Multi-file batches**: one offer/prompt for N files (≤32), one folder pick, files run sequentially with one aggregate card (first failure aborts the rest). **Auto-accept trust list** ("Always accept" checkbox on the prompt; managed in Settings → App → File sharing, master toggle): trusted senders skip all dialogs and save to Downloads (collision-suffixed), with OS notifications for auto-starts and for offers arriving while unfocused. Floating bottom-right progress cards (rate, i/N, cancel, "Show in folder"). 2 GB+ files with flat renderer memory; multiple concurrent transfers.
- **P2P Soundboard:** a **Soundboard** button next to React triggers bundled preset SFX or short custom local clips — everyone in the room hears it instantly, mixed through the shared master bus, never routed through the mic. Custom clips are added by dropping audio into a watched inbox folder or via Settings → Soundboard → "Add Sound"; a bundled ffmpeg binary trims to 5s, loudness-normalizes, and compresses to 128kbps MP3 (content-hash-addressed for dedup — **not** Ogg Vorbis, which embeds a per-encode random serial number that breaks hash-based identity). Custom clips **sync automatically in the background** to room peers over a silent, ephemeral WebRTC pull that reuses the file-transfer DataChannel machinery unmodified (no accept prompt — the whole feature is opt-out, not opt-in-per-transfer). One master toggle in Settings → Soundboard (default **off**) covers the button, the watcher, and all sync/playback; separate toggles there let a user hide the bundled presets (custom clips only) or mute all inbound clips from other room members by default.
- **Shell:** frameless 3-zone lounge (sidebar + adaptive Voice-Lounge/Gallery/Theater stage + control bar) under a title bar; system tray; taskbar badges + desktop notifications for unread chat when unfocused; compact sidebar-dock mode; drag-resizable sidebar + chat; light/dark themes; custom SVG room icons. Settings persist to Electron `userData`.
- **Packaging:** Windows builds via electron-builder — a portable single `.exe` and an NSIS installer (classic wizard: pick install directory, Start Menu + Desktop shortcuts, per-user by default so no admin/UAC prompt), both produced from one `npm run dist`.
- **Auto-update (installer builds only):** `electron-updater` checks GitHub Releases on this repo — silently at startup (10s delay) + every 4h, never the portable build (no installed app for it to replace — gated on `PORTABLE_EXECUTABLE_FILE`). Prompt-before-download: `update-available` surfaces a small floating card ("Update available — Download?"); only a click starts `downloadUpdate()`. Downloaded → "Restart & Update" (or installs silently on next quit, `autoInstallOnAppQuit`). A manual "Check for Updates" button lives in the title-bar → About dialog and always re-checks (bypasses the background poller's once-found guard), with feedback states (checking/up-to-date/error) the silent path stays quiet about. Releases ship via the tag-triggered `.github/workflows/release.yml` (push a `v<version>` tag matching `apps/desktop/package.json`) — `npm run release` runs the equivalent locally.

**Not yet built / deferred:** code-signing (users get a SmartScreen warning on both the portable `.exe` and the installer — the downloaded update installer too), macOS/Linux builds, and friends "invite to room" (needs a presence channel beyond the in-room model).

## Signaling protocol (`packages/shared/src/protocol.ts`)

`ClientMessage` (client→server) and `ServerMessage` (server→client) are **separate discriminated unions** (the server stamps `from`; clients only set `to`). Relayed WebRTC messages: `offer`/`answer`/`ice-candidate`, plus `relink` — the media-mesh recovery teardown request ("close your link to me; my fresh offer follows"), needed because a recreated `RTCPeerConnection` mints a new DTLS certificate that the remote's OLD pc can never accept, so both sides must rebuild. (Shared is split by domain — `capacity.ts`, `sanitize.ts`, `protocol.ts`, `settings.ts`, `ice.ts` — all flat-re-exported from `src/index.ts`, so consumers keep importing from `@chickadee/shared`.)

### Pattern 1: per-peer mirror broadcasts

Room-wide broadcasts follow one **mirror pattern** — adding a new per-peer state means touching the same five places:

1. add a field to `Peer` + a `*-state` message to both unions (`packages/shared/src/protocol.ts`)
2. init the field (`handlers/room.ts handleJoin`) + add a `handle*State` broadcast in `apps/signaling/src/handlers/mirrors.ts`, dispatched from `src/server.ts`
3. add a `case` in `applyPresenceUpdate` (`useSignaling.ts`) to update `peers`
4. send it from `usePeerMesh.ts` / `App.tsx` on the local toggle
5. react to it in the mesh `subscribe` handler if it affects media

Examples: `muted` (mic-state), `cameraOn` (cam-state), `screenStreamId` (screen-state), `speaking` (speaking-state), `avatarDataUrl` (avatar-state, space-wide), `voicePreference` (voice-state, room-only), `accentColor` (accent-state, space-wide), `soundboardClips` (soundboard-manifest-state, space-wide).

**Space-wide vs room-wide:** most mirror messages broadcast to the room only (`broadcast`); to reach all space members regardless of room, also call `broadcastSpace` (as `handleStatusState`/`handleAvatarState` do). The client applies room-level updates via `applyPresenceUpdate` (updates `peers[]`) and space-level updates via the `space-peer-update` case (updates `spacePresence[]`).

### Pattern 2: server-arbitrated single slot (stage spotlight)

**Stage spotlight is a server-arbitrated single slot, NOT a per-peer mirror.** It follows the `room-full` reject pattern, not the 5-place mirror. The server keeps a `spotlights` map (`RoomId → { holderId, kind }`). `claim-spotlight {kind, force}` grants if free / held-by-you / `force`, then `broadcast`s `spotlight-state {holderId, kind}` to the **whole room** (incl. the claimant so it confirms); a blocked non-`force` claim replies `spotlight-busy {holderId}` (only to the claimant → drives the take-over prompt). `release-spotlight` + disconnect/room-leave clear it (broadcast `spotlight-state {null, null}`). `welcome` carries the current holder for mid-joiners. Server-side the arbitration decision is the pure `evaluateSpotlightClaim` in `apps/signaling/src/logic.ts` (unit-tested); the handlers live in `handlers/spotlight.ts`. Client state lives in `useSignaling` (`spotlightHolderId`/`spotlightKind`, handled in `applyPresenceUpdate`); `hooks/useStageSpotlight.ts` owns the actions (auto-claim on screen-share, manual camera spotlight, take-over prompt, reconnect re-claim via `desiredStageKindRef`), and App renders the take-over modal from its `pendingTakeover`.

### Pattern 3: directed space-scoped relay (file transfer)

**File transfer is a directed, SPACE-scoped relay** (`file-offer`/`file-answer`/`file-signal`/`file-cancel`) — not the room-scoped `relay()` and not the 5-place mirror. `handlers/fileTransfer.ts relayFileMessage` looks the target up in `spaceConnections` by **session id** so a transfer can reach a peer in another room (the USERS list is space-wide); the server sanitizes `file-offer` name/size (`sanitizeFileOfferMeta`) — and, for batches, the whole `files` array (`sanitizeFileOfferFiles`: 2..`MAX_BATCH_FILES` entries, one bad entry drops the offer) — clamps `file-cancel` reason, relays `file-signal` SDP/ICE verbatim, and tracks nothing (stateless). **Batches are an orchestration-layer concept:** `file-offer.files` present ⇒ `transferId` is the batch id; per-file transfers use DERIVED ids `` `${batchId}:${index}` `` (`fileTransferPolicy.makeBatchFileId`) that only ever appear in `file-signal`, while `file-answer`/`file-cancel` always carry the root id (batch-wide semantics). **Don't reuse the `offer`/`answer` message types for new WebRTC consumers** — `usePeerMesh` treats any inbound `offer` as a media-mesh link.

The soundboard clip-fetch handshake (`soundboard-fetch-request`/`-signal`/`-cancel`, `handlers/soundboard.ts relaySoundboardFetchMessage`) reuses this **exact** relay shape — new message types, not the `file-*` ones (`useFileTransfers` pattern-matches on those exact strings). It differs only in orchestration, not protocol: there's no `-answer` (a possessor silently opens send links for whatever it has, or replies `-cancel`), and `-signal`'s `requestId` field always carries the DERIVED id `` `${requestId}:${index}` `` — always the derived form, even for a single-hash request, since (unlike file transfer) there's no user-facing single-vs-batch UX to preserve. See "Soundboard" under Key conventions & gotchas for the full client-side picture.

### Pattern 4: server-authorized commands (moderation)

**Moderation is server-authorized commands over sticky space state + derived room state** (`handlers/moderation.ts`). Every command (`kick-user`/`ban-user`/`unban-user`/`set-room-lock`/`set-space-lock`/`transfer-ownership`/`seed-moderation`) opens with the pure `canModerate` matrix in `logic.ts` (unit-tested; silent no-op on deny, extending the `handleSetBanner` precedent — a deterrent, not a security boundary). Key shapes:

- **Room Moderator is DERIVED, never stored:** `roomModeratorId(roomId)` = first entry of the room's insertion-ordered members Map (= longest-present). A join can't change it, so only the two leave paths call `syncRoomModerator` (broadcasts `moderator-state` room-wide); `welcome` carries `moderatorId` in **both** welcome sites. The owner *outranks* the mod (`isOwner || isModerator`) rather than displacing the slot. A mod's reconnect blip passes the role on — accepted.
- **Sticky space maps, re-seeded after restarts:** `spaceBans` (userId → displayName-at-ban-time) and `spaceLocks` survive `scheduleSpaceCleanup` like `spaceOwners`; all of them vanish only on a full server restart. Every member persists `SpaceInfo.bannedUsers`/`locked` (from space-wide `ban-state`/`space-lock-state`/`welcome` fields), and the client that a connection confirms as owner sends one idempotent `seed-moderation` (server adopts each part only if it has **no** record — `shouldAdoptSeed`). Bans key on the client-asserted `userId` (evadable by identity reset — accepted); banning also evicts the target from `spacePresence` so they drop off the locked-space allowlist. `roomLocks` is **ephemeral** (composite ids, deleted in the two room-empty paths next to `spotlights.delete`; only ACTIVE rooms can be locked).
- **Join gates run in `handleJoin` BEFORE ghost cleanup** (`evaluateJoinGate`: ban beats lock; a locked space admits the owner + anyone in `spacePresence`, i.e. members and <10-min reconnects) and in both room-entry paths (`evaluateRoomEntry`: lock beats full; owner bypasses the lock, never the 8-cap). Rejects are `join-denied {reason}` (room-full pattern; terminal client-side as status `'kicked'`).
- **Kick semantics:** room-kick = `kicked {scope:'room'}` then the existing `handleJoinRoom(target, null)` lobby-return (space connection survives; client `leaveRoom()` + toast). Space-kick/ban = `kicked {scope:'space', reason}` then a normal `socket.close(4000)` (listeners intact so `handleDisconnect` runs — unlike ghost cleanup); terminal client-side, no auto-reconnect.
- **Client:** state in `useSignaling` (`moderatorId`/`lockedRooms`/`spaceLocked`/`bannedUsers`, welcome merges space fields omit-means-keep); UI = `RoleStar` (gold owner / silver mod; gold suppresses silver) in `FriendRow` + `ParticipantTile`, right-click `UserContextMenu` (items mirror `canModerate`; hidden entirely when unauthorized), lock items in `RoomContextMenu` and the space banner's action row (`SpaceSwitcher`), and the owner-gated Moderation section in `SpaceSettingsModal` (lock/transfer/ban-list; live-gated to the connected space). `joinRoom()` pre-blocks locked rooms with a transient toast since the server reject is terminal.
- **Room/space governance (same ownership, WHOLE-LIST validation):** `update-rooms` stays a whole-list replace but is now diffed server-side by stable room id (`evaluateRoomsUpdate` in `logic.ts`, pure + unit-tested) against: owner = anything; member = rename/remove only rooms where `Room.createdBy === userId`, and ≤1 created room after the update (delete-then-recreate in one update is legal). Surviving rooms keep their CURRENT `createdBy` (anti-tamper); adds are stamped with the sender server-side; legacy/unstamped rooms (incl. `DEFAULT_ROOMS`) are owner-managed and count against nobody's quota. `handleUpdateRooms` keys on `conn.space` (**the wire `spaceId` is ignored** — a client can't rewrite another space's rooms); a denied update resyncs ONLY the sender with the authoritative `rooms-updated` so their optimistic local write reverts. `rename-space` is owner-gated (silent no-op). An unowned space is **strict until claimed** — no owner means nobody passes the owner checks (members can still create their one room). `createdBy` persists in every member's room list and re-seeds on join (`sanitizeRoomList` guards both the seed and updates, `MAX_ROOMS` cap), so governance survives server restarts. Client: `canManageRoom` gates `RoomContextMenu` Rename/Remove (menu doesn't open with zero items), the sidebar "+" pre-checks the one-room quota with a toast, `SpaceSettingsModal` disables the name field for non-owners.

## Key conventions & gotchas

### Connection & space lifecycle

- **`signaling.join` is NOT idempotent.** Every call does `closeSocket()` → reset state to `connecting` → `connect()` (new socket → new `selfId` → `usePeerMesh` rebuilds the whole WebRTC mesh). So the space-level connection effect in `App.tsx` must depend on the **current space's** connection params (`activeSpaceSignalingUrl` / `activeSpaceJoinSecret`, derived just above it), **never the whole `spaces` array** — otherwise editing/deleting *another* space reconnects the active call. (`useSpaces` calls `setSpaces` on space CRUD but not on room edits, so room changes are safe.)
- **Space existence is liveness-based, gated only on new invite-code joins.** The server is in-memory, so a Space "exists" only while ≥1 member is connected (`spaceConnections.has(spaceId)`); it's resurrected from a joining client's local room list each time. A non-mutating `check-space` → `space-status` probe validates existence without joining (honors `CHICKADEE_JOIN_SECRET`). Only **brand-new invite-code joins** are gated (`useSpaces.addSpace`/`initFirstSpace` `await verifySpace` over a throwaway socket → `'exists' | 'not-found' | 'unreachable'`). **Known/locally-persisted Spaces (sidebar switch, reconnect) are never gated** — local persistence (`store.getSpaces()`) is what makes Created Spaces durable. `verifySpace` (`lib/verifySpace.ts`, exposed through `useSignaling`) uses its own temp `WebSocket`, never `socketRef`, so it can't disturb the persistent connection.
- **Resilience:** `useSignaling` auto-reconnects (equal-jitter backoff, `reconnecting` status) and re-joins after a drop; an app-level `ping`/`pong` heartbeat detects half-open sockets (the backoff curve + timeouts are pure in `lib/reconnectPolicy.ts`, unit-tested; the server runs its own 10s ws-level heartbeat so survivors get `peer-left` for a silently-dead peer in 10–20s). On the re-`welcome` (new `selfId`) `usePeerMesh` rebuilds all links and re-announces mic/cam/screen state; `peerLink` calls `restartIce()` on `failed`/`disconnected`. Reconnect is *not* a terminal status, so local media survives the blip.
- **Per-link health monitor (`webrtc/connectionHealthPolicy.ts`, pure + unit-tested; the one interval in `usePeerMesh` — setInterval, never rAF).** Every 2s each link's `getStats()` inbound-audio packet count feeds the pure `evaluateHealth` reducer. Packet advance is the ground truth (`connectionState` can claim `'connected'` with dead RTP; a `'connecting'` label can lag while media flows): a once-live link whose packets freeze 6s escalates restartIce → verified-by-stats → **scoped relink** (tear down + rebuild ONE pair via the `relink` message — never the whole mesh) → after 3 failed relinks, `'failed'`. A link stuck in `new`/`connecting` 10s (offer/answer lost in the fire-and-forget relay — the "no audio on join" bug) relinks directly. Opus DTX means even a MUTED peer emits comfort-noise packets (~2.5/s), so zero-delta is transport death, never silence; a link that has never received (peer with no mic sender) is exempted via `hasReceived`. Key invariants: the health entry **survives relinks** (attempt cap can't reset) but its packet baseline resets to 0 (fresh pc counters restart); the relink RECEIVER resets baseline + `createdAt` in the `case 'relink'` handler (it never runs the reducer's relink transition). A `reconcileMesh` backstop (10s grace — mesh listeners see messages before presence state updates) closes links whose peer vanished and rebuilds peers left linkless. UI: `RemoteMedia.health` (`'ok' | 'recovering' | 'failed'`, patched only on transitions) drives the tile's existing `.tile__conn` pill and outranks the raw `connectionState` label.
- **Spaces vs rooms — composite room IDs on the server.** The in-memory key is `${spaceId}:${roomId}`. The `join` message carries `spaceId`; clients only see bare `roomId` strings. `spacePresence`/`spaceConnections` track all peers in a space regardless of room (for the friends sidebar). Space metadata clears server-side when the last peer leaves.
- **Rooms are unified `'hybrid'` (8-cap) — `capacityForType()` returns 8 for every type.** The old `voice`(8)/`video`(4) split is gone; `RoomType` keeps `'voice'|'video'|'hybrid'` only for wire back-compat, and `normalizeRoomType()` is applied **server-side only** (`sanitizeRoomList` — old-build clients may still send legacy types; clients never migrate persisted state). `Room.type` no longer drives any behavior (single sidebar ROOMS list, no type picker, no `allowVideo` gate — video controls always shown). New rooms are created `type: 'hybrid'`. Sidebar collapse is one `roomsSectionCollapsed` flag (was two).
- **`useSpaces` is the single writer for room persistence.** All room list mutations (`createRoom`/`renameRoom`/`removeRoom` + the `rooms-updated` sync effect) route through `useSpaces.updateRooms(next)`, which calls both `setRooms` (state) and `store.setRooms` (disk). Don't call `store.setRooms` directly from `App.tsx`.

### Deployment, ICE & hardening

- **Packaged app defaults to the hosted signaling server** (`wss://chickadee-signaling.onrender.com`, chosen via `app.isPackaged` in `buildConfig`); overridable via env or a `.env` beside the exe. No in-app field for it yet.
- **Signaling in prod runs on Bun** — the `oven/bun` Docker image runs `apps/signaling/src/index.ts` directly (no `tsc`/build; Bun resolves shared's `.ts` entry). `apps/signaling/Dockerfile` builds from the **repo root** context. The Render service **auto-deploys from `main` on every push** — merging to `main` IS a production deploy of the signaling server (branches: just `main` + `dev`; the old `release` deploy-gate branch is deleted).
- **ICE: STUN-only by default, TURN opt-in.** `STUN_SERVERS` (two independent providers — Google + Cloudflare — so one provider's outage can't sink candidate gathering) always; TURN only from `CHICKADEE_TURN_*` env. The old free public TURN fallback (OpenRelay) was defunct and is **deleted** — it cost 3 doomed relay allocations per negotiation for zero benefit; don't reintroduce a public TURN default. Both-sides-symmetric-NAT play needs a self-hosted TURN (see README).
- **Signaling hardening + access model.** The server is **open by default** (any client with a non-secret `spaceId` can join). It validates/clamps every client field via shared helpers (`packages/shared/src/sanitize.ts` — `sanitizeAvatarDataUrl` 256 KB + image-mime, `clampString`, `sanitizeStatus`, `sanitizeAccentColor`; all unit-tested), sets a WS `maxPayload`, and rate-limits per connection (200 msg/s; the window math is pure in `apps/signaling/src/logic.ts`). Two server-side env vars lock down private deployments: `CHICKADEE_JOIN_SECRET` (clients send a matching `secret` in `join`) and `CHICKADEE_ALLOWED_ORIGINS` (empty = allow all, since the Electron client sends no Origin). Avatars are re-validated client-side before any `<img src>`.

### Electron shell & config

- **Preload module format is load-bearing.** `apps/desktop/package.json` is **NOT** `"type": "module"`, so electron-vite emits **CommonJS** `out/main/index.js` + `out/preload/index.js`, and main loads `../preload/index.js`. If the package becomes ESM the preload emits `index.mjs`, the path silently mismatches, and **`window.chickadee` becomes `undefined`** (→ blank screen). Keep main's preload path and the build output extension in sync.
- **npm 11 gates install scripts (`allowScripts`).** A fresh `npm install` would otherwise skip Electron's postinstall (downloads `electron.exe`) and `uiohook-napi`'s native build, leaving the app unrunnable. Root `package.json` carries a **version-pinned** `allowScripts` allow-list so install runs them. **When you bump Electron or uiohook-napi, re-approve** via `npm approve-scripts <pkg> --allow-scripts-pin`. (Agent shells set `ELECTRON_RUN_AS_NODE=1`; clear it with `env -u ELECTRON_RUN_AS_NODE` to launch Electron rather than run it as Node.)
- **Renderer runs with `sandbox: true`** (`main/index.ts`). The preload uses only sandbox-safe APIs (`contextBridge`/`ipcRenderer`/`webFrame`) and reads config from `process.argv` (`additionalArguments`); `@chickadee/shared` is bundled in (pure TS, no node builtins). Adding a node builtin to the preload breaks it under the sandbox.
- **Runtime config flows main → preload in two channels.** `buildConfig()` reads env (`CHICKADEE_SIGNALING_URL`, `CHICKADEE_TURN_*`; `loadDotEnv()` walks up for a `.env`) and JSON-passes only the small fixed `{ signalingUrl, iceServers, appVersion }` on `--chickadee-config=` argv. **Persisted `settings` is delivered separately over the synchronous `chickadee:get-settings` IPC**, NOT argv — settings include the base64 `avatarDataUrl` and argv has a ~32 KB Windows limit that a larger avatar would overflow → silent fallback to `defaultSettings()`. Both arrive synchronously so `lib/settings.ts` can read `window.chickadee.settings` at module load. `externalizeDepsPlugin({ exclude: ['@chickadee/shared'] })` bundles shared into main/preload.
- **Renderer accesses `window.chickadee` defensively** (optional chaining + guards); the `ErrorBoundary` surfaces any renderer throw instead of blanking.
- **Frameless window.** `BrowserWindow({ frame: false })`; the `TitleBar` is the sole drag region (`-webkit-app-region: drag`) with `no-drag` on its interactive children (window controls, chat pill, compact-collapse button) — the space banner and room content are plain, non-draggable surfaces; `window.chickadee.windowControls` → IPC → minimize/maximize/close. **Entry flow:** no join form — a first-run `NameModal` (name persisted in `userData`), then clicking a sidebar room calls `signaling.join(...)`. Sidebar room counts are only known for the room you're in. **PTT keys are captured system-wide — pick a key not used in-game.**
- **`win.setResizable()` resets min/max size on Windows — never call it to enforce a width cap.** The compact-dock cap (`COMPACT_MAX_WIDTH` 520) is enforced via `setMaximumSize` + the constructor `maxWidth`, never `setResizable` (calling it after `setMaximumSize` silently wipes the constraints → the sidebar stretches infinitely). The window is created `resizable: true` once; compact handlers only `setMinimumSize`/`setMaximumSize`. Belt-and-suspenders: a `will-resize` listener re-clamps width to `clampCompactWidth` while compact (catches Aero snap / double-click-maximize); the compact width IPCs (`window-set-compact`, `window-set-width`) clamp too.
- **Dev `userData` is per instance-slot** (`temp/chickadee-dev-<CHICKADEE_INSTANCE|0>`, unpackaged only): a fixed dir so settings persist across restarts, while a 2nd test instance (`CHICKADEE_INSTANCE=1`) stays isolated.
- **Beta clean-baseline policy: never add read-time schema migrations.** Persisted-shape changes (settings keys, room/space shapes, the soundboard manifest) are handled by `main/versionGate.ts`, which wipes ALL local state (session data, `settings.json`, soundboard manifest/cache/inbox — including `userId` and the joined-Spaces list, accepted for beta) on any `app.getVersion()` change. When you change a persisted schema, **bump `apps/desktop/package.json` `version`** to arm the wipe — do not add coercion/patch code on the read path. Readers validate against the CURRENT schema only and fall back to `defaultSettings()`/drop-the-entry for anything missing or invalid. (Runs in dev too — the dev temp userData resets on version bumps.)
- **Orphaned signaling server on `:8080`:** `concurrently` on Windows doesn't reliably kill the `bun --watch` child on Ctrl-C, so a stale server can keep `:8080` bound. The root `predev` script (`node scripts/free-port.mjs 8080`) frees it before each `dev`; run it manually if you hit the port outside `npm run dev`.
- **electron-updater (`main/autoUpdate.ts`) is a no-op unless `app.isPackaged` AND not portable.** Portable-mode detection is `process.env.PORTABLE_EXECUTABLE_FILE` (set by electron-builder's portable launcher) — there's no installed app for the updater to replace, so it must never run there. `electron-builder.yml`'s `publish` block (GitHub, this repo) is what lets electron-updater resolve `latest.yml` from Releases; `apps/desktop/package.json` `version` is the version compared, so a release's tag should match it. electron-builder's GitHub publisher reads the `GH_TOKEN` env var specifically, **not** `GITHUB_TOKEN` — `.github/workflows/release.yml` maps `secrets.GITHUB_TOKEN` into it.
- **electron-builder's `releaseType: release` publish option is unreliable — verify every release isn't left as a draft.** Observed firsthand shipping 0.4.2: even with `releaseType: release` set, the GitHub release still came back `draft: true` (electron-updater's GitHub provider silently ignores drafts, so this fails invisibly — no build error, just an update nobody ever receives). `.github/workflows/release.yml` now has an explicit `gh release edit "$tag" --draft=false` step after the build — don't remove it trusting the electron-builder config alone. A local `npm run release` needs the same manual follow-up.
- **The GitHub repo must stay public, or auto-update breaks invisibly for everyone.** electron-updater's HTTP client makes anonymous, unauthenticated requests when checking for updates (it never ships a token in the packaged app — embedding one would let anyone who extracts it from the `.exe` use it, defeating any privacy purpose). Against a private repo, GitHub 404s those requests (it masks the real 401/403 to avoid confirming a private repo's existence), surfacing as a raw, unhelpful error via `chickadee:update-error` ("Please double check that your authentication token is correct..." — a hard-coded electron-updater string for any 404, not specific to real auth problems). Confirmed by making the repo private→public and re-testing: `curl` to `.../releases/latest/download/latest.yml` must return 200 anonymously for the update flow to function at all.

### Media pipeline

- **Screen vs camera classification.** A sharer broadcasts its screen `MediaStream.id` via `screen-state`; the receiver matches incoming `ontrack` streams against it (`recomputeRemote` in `usePeerMesh.ts`), robust to message/track ordering (`welcome` carries `screenStreamId` for mid-share joiners). The id-matching decision is the pure, unit-tested `classifyPeerStreams` in `webrtc/meshLogic.ts` (last non-screen id wins); `recomputeRemote` looks the `MediaStream` objects back up so the `cameraStream` reference stays stable.
- **Asymmetric encoding = the mesh-safety lever for 8-cap rooms (`webrtc/encodingParams.ts`, pure + unit-tested).** A `VideoRole` = `'stage' | 'thumbnail'`: exactly one stream per room is `stage` (the spotlighted screen/camera — tier-capped, high quality), every other webcam is a fixed tiny `thumbnail` (~200 kbps + `scaleResolutionDownBy 3` + 15 fps, any tier). The stage is further clamped by `applyUploadBudget(enc, viewers, budget) = min(tierCap, budget/viewers)` — total stage upload stays ≈`budget` no matter how many watch (per-viewer quality degrades as the room fills). Budget = the user's **Upload limit** setting (`uploadBudgetMbps`, Mbps → bps; `0` = unlimited → tier cap only; default 12 Mbps, `STAGE_UPLOAD_BUDGET_BPS`). `usePeerMesh` gets `myStageKind`/`selfWatcherCount`/`uploadBudgetBps` from `App.tsx` and re-applies caps to every sender via the existing `applyEncoding()` (no renegotiation). **No simulcast** — each sender has one role at a time, so the single-`encodings[0]` path in `peerLink.applyVideoEncoding` is unchanged. Extend caps *here*, not in `peerLink`.
- **Stage-driven layout — spotlight is the single source of truth (`App.tsx`).** `theater = spotlightHolderId != null` — the whole stage derivation (holder/subscription/stream-kind) is the pure `selectStage` in `lib/stageSelection.ts` (unit-tested); App resolves its ids back to live objects. `theater` → `.presentation` (`ScreenView` stage tile — now renders a screen OR camera via its `kind` prop — + `.filmstrip` of everyone). Else `.grid` (Voice Lounge with 0 videos / Gallery with videos — same container; `ParticipantTile` decides avatar-vs-video per tile). The `tiles` fragment is shared between grid and filmstrip with stable `key={peer.id}` so React reconciles across the transition. Grid CSS is styled for up to 9 tiles. **Viewing stays opt-in even for the stage** (`videoSubscriptions`/"Watch"): a non-subscriber sees a large "Watch [name]" placeholder on the stage, so non-watchers never pull the stream. Screen shares auto-claim the spotlight on start (App effect on `mesh.sharingScreen`); cameras spotlight via the `.tile__spotlight` button. **Do not** reintroduce the old screen-only `presenting`/`activeScreens` — layout is `spotlightHolderId`-driven now.
- **Screen capture path — keep the custom in-app picker.** `desktopCapturer` (main) lists sources over IPC; capture uses `getDisplayMedia()` fulfilled by `setDisplayMediaRequestHandler` in main, with `audio: 'loopback'` for Windows system audio. We use `ScreenSharePicker.tsx` (thumbnails + audio + resolution/fps) with `useSystemPicker: false`. **Don't migrate to Electron's native `useSystemPicker`** — it's macOS-15+-only and experimental (toggle-hang bug electron/electron#45306); on Windows the loopback handler already *is* the native system-audio path.
- **Screen-share audio excludes our own voices via `restrictOwnAudio`** (`usePeerMesh.startScreenShare`). Windows loopback captures the entire output mix — including incoming peer voices we play locally — which made remote peers hear themselves. Fix: request `audio: { restrictOwnAudio: true }` (feature-detected; Chromium 141+ / shipped here), which drops audio originating from our own document while keeping other processes' (the game's). **Do not** route peer voices off `ctx.destination` to "fix" this — `restrictOwnAudio` is the supported path.
- **File transfers ride a dedicated `RTCPeerConnection` per transfer (`webrtc/fileTransferLink.ts`) — never the media mesh's `peerLink`.** Sender = offerer, fixed roles (no perfect negotiation), no ICE restart (fail fast + retry); an open DataChannel survives room switches, mesh rebuilds, and signaling reconnects, which is the point. Wire protocol: binary frames = ordered 64 KiB chunks (offsets implicit), string frames = JSON control (`{t:'pause'|'resume'|'done'|'ack-done'|'cancel'|'error'}`, parsed by `fileTransferPolicy.parseControlMessage`). Two flow controls, both required: sender pauses on `dc.bufferedAmount` ≥ 8 MiB (resumes via `bufferedAmountLowThreshold`), receiver sends `pause`/`resume` when its disk queue (`webrtc/transferQueue.ts`) crosses 16/4 MiB — that bound is what keeps a 2 GB receive at flat memory. `done`→`ack-done` matters: the sender's last `send()` only proves local buffering; only `ack-done` (sent after the `.part` rename) completes the card. Constants + pure decisions live in `webrtc/fileTransferPolicy.ts` (unit-tested) — tune there, not inline.
- **Receiver disk IO lives in main (`main/fileTransfer.ts`), never the renderer.** The sandboxed renderer moves opaque `transferId`s + `Uint8Array` chunks over IPC (`chickadee:begin-file-save` → Save dialog + `.part` write stream, `write-file-chunk` — the awaited invoke IS the backpressure, `end-file-save` → rename, `abort-file-save` → unlink). Batches and the trust list add destination channels around the SAME byte path (batch files land in the same `saves` map under derived ids): `begin-batch-save` (ONE `showOpenDialog` folder pick authorizes `batchDirs`), `begin-batch-file-save` (opens one file's stream in that dir, lazily on first write — zero-byte files open at finish), `begin-auto-save`/`authorize-auto-batch` (dialog-less Downloads for trusted senders), `release-batch`. All auto-chosen names are collision-suffixed Explorer-style (`suffixedFileName`, checks target + `.part` + active saves — two identical names in one batch just suffix). The final filename only ever appears via the end-save rename, so a torn transfer can't masquerade as the finished file; `will-quit` destroys open streams + batch state. `show-file-in-folder` takes the transferId (a batch id resolves to its first finished file) — the renderer never handles filesystem paths. Orchestration is `hooks/useFileTransfers.ts` (offer FIFO → one accept modal at a time, which also serializes the Save/folder dialogs; batch receive links are created lazily on each file's first `file-signal`, sender runs files strictly sequentially, first failure aborts the remainder; auto-accept falls back to the manual prompt if the dialog-less save can't start — never a silent decline; handshake messages address session ids captured at offer time, so a mid-handshake reconnect times out by design — an accepted limitation).
- **Drag-and-drop + notifications hardening (easy to regress):** App installs window-level `dragover`/`drop` `preventDefault` and main installs a `will-navigate` guard — without BOTH, a file drop that misses a USERS row navigates the window to `file://`. Renderer OS toasts (`lib/notify.ts`) only work because `'notifications'` is in `GRANTED_PERMISSIONS` (`main/index.ts`) — the permission handler is deny-by-default, so removing it silently kills them.

### Audio engine

- **Shared `AudioContext` — never close it in `usePeerMesh`.** `lib/audioContext.ts` holds the one `AudioContext` shared between `sfx.ts` and `usePeerMesh`. On teardown, `usePeerMesh` disconnects its `GainNode`/`AnalyserNode` but **must not call `audioCtx.close()`** — SFX needs the context to survive join/leave cycles.
- **Master output bus — connect locally-played audio to `getMasterBus()`, not `ctx.destination`.** `lib/audioContext.ts` exposes `getMasterBus()`, a single brick-wall limiter (`DynamicsCompressorNode`, −1 dBFS, ratio 20) wired once to `ctx.destination`. Everything heard locally (per-peer playback, SFX) connects to it so stacked gains + multiple talkers can't clip. The mic graph (→ `MediaStreamDestination` for WebRTC) and analyser taps intentionally bypass it. The output device is a single property of this one context: set once via `setOutputSink()` (one App effect on `outputDeviceId`), not per-tile.
- **Per-peer incoming-audio graph — use `createMediaStreamSource`, NOT `createMediaElementSource`.** `ParticipantTile` (via `usePeerAudioGraph`) routes each remote peer's audio through the shared context (for >100% gain + the normalize compressor) by sourcing from the **stream**, then muting the `<video>`. `createMediaElementSource` (1) **permanently binds** the element, so React `StrictMode`'s remount throws "already connected", and (2) yields **silence** for remote WebRTC streams in Chromium. The graph rebuild depends on `[cameraStream, isSelf, normalize]`; `cameraStream` identity is stable across camera on/off (`recomputeRemote` reuses the same object). Fallback if no `AudioContext`: `audioRouted` stays false, the `<video>` stays unmuted, and a separate effect applies `el.volume` (0–1) so per-peer volume + Deafen still work (no >100% boost).
- **Per-tile silence = volume 0 (no separate mute state).** The per-tile volume icon toggles silence by setting that peer's volume to `0`, restoring the prior level via a session `lastNonZeroVolumeRef` (per `peer.id`) in `hooks/usePeerVolumes.ts`. It routes through `handleVolumeChange`, so silence persists by `userId`. There is intentionally **no** per-peer mute flag — `pvMuted` is just `volume <= 0`.
- **Screen-share audio has its own independent per-peer volume, separate from voice.** `usePeerVolumes.ts` exports a second hook, `usePeerScreenVolumes`, built on the same generic dual-keyed control (`usePeerVolumeControl`, DRY'd out of the original single-purpose hook) but backed by its own persisted key (`peerScreenVolumes`, not `peerVolumes`) — a peer's game/system audio can be muted or boosted 0–200% without touching their mic. The gain-graph plumbing itself (`createMediaStreamSource` → optional compressor → `GainNode` → `getMasterBus()`, live-gain updates, no-`AudioContext` fallback) was extracted from `usePeerAudioGraph` into stream-agnostic `hooks/useAudioGraph.ts` so `ScreenView` could reuse it for a remote screen share's audio track; `usePeerAudioGraph` is now a thin composition of its own `<video>`/srcObject binding + a call into it. A spotlighted **camera** has no separate audio track (the person's mic *is* the audio, already fully handled by their always-rendered filmstrip tile), so `ScreenView` never builds a second graph for that kind — its `<video>` element is now **unconditionally muted for every remote kind** (previously only self was muted, which meant a spotlighted remote camera's mic voice played twice: once via the filmstrip's graph, once raw through the stage's own unmuted `<video>` — fixed as part of this change, along with removing the now-dead per-element `setSinkId` call it depended on).
- **Audio-decision loops run on `setInterval(~20 ms)`, NOT `requestAnimationFrame`.** `useVoiceActivation` reads an `AnalyserNode` + RMS on a fixed ~50 Hz timer (`COMPUTE_INTERVAL_MS`, timestamped via `performance.now()`). **It must NOT use rAF:** rAF stalls to near-zero when the window is **minimized**, which froze the VAD and dropped background voice. `setInterval` keeps firing at full rate while minimized because `backgroundThrottling:false` protects timers (the same reason PTT/TTS work minimized). The cadence also caps analyser work below the display refresh rate. Gate math is timestamp-based, so cadence affects responsiveness, not correctness.

### Soundboard

- **Presets need no sync; only custom clips do.** Presets are bundled assets (`assets/soundboard-presets/*.mp3`, discovered via the same `import.meta.glob` pattern as `room-icons/` but with `?url` instead of `?raw` since they're binary, not inlineable text) — same build, same bytes, same hash on every peer, so they're playable with zero network activity. Only user-added custom clips ever need P2P distribution.
- **Custom-clip identity is content-hash-addressed, and the codec choice is load-bearing because of it: MP3, not Ogg Vorbis.** Empirically verified (transcode identical input twice, diff the output) that ffmpeg's Ogg muxer embeds a randomized per-encode stream serial number, so Vorbis output is **not byte-deterministic** even for byte-identical input — this silently breaks hash-based dedup (re-adding the same clip mints a new hash every time; two peers with "the same" sound never recognize it as such). `-serial_offset` does not fix this (still randomized around the offset). MP3 (`libmp3lame`) elementary streams have no such per-file randomness and were confirmed bit-identical across repeated encodes of the same input. **If the codec is ever revisited, re-run this determinism check before switching** — don't trust codec docs alone, verify actual output bytes.
- **Ingest pipeline (`main/soundboardTranscode.ts` + `main/soundboardLibrary.ts`).** A bundled `ffmpeg-static` binary (added to `dependencies`, not `devDependencies` — electron-builder prunes those — plus an `asarUnpack` entry in `electron-builder.yml` mirroring `uiohook-napi`'s, since you can't spawn an exe from inside asar) runs a fixed argv: `-t 5` is an **input-side** trim (before `-i`), so the demuxer never reads past 5s and the downstream `dynaudnorm` filter physically cannot see discarded audio; loudness normalization is single-pass `dynaudnorm` (not two-pass `loudnorm` — EBU R128 accuracy isn't worth a second ffmpeg spawn for a ≤5s clip); `-progress pipe:1` gives progress **and** the clip's duration on completion, avoiding a second bundled binary (ffprobe). `main/soundboardLibrary.ts` owns an `fs.watch`-ed inbox folder (`userData/soundboard-inbox/` — one ingest path whether a file arrives via the in-app "Add Sound" picker or a direct Explorer drag) with a poll-based stabilize/debounce (`isSizeStable`: 3 consecutive matching size samples) before touching a file with ffmpeg, and a content-addressed cache (`userData/soundboard-cache/<hash>.mp3`) shared by local-ingest output and P2P-synced clips alike. Pure decision logic (the debounce predicate, the source-extension allowlist, cosmetic name derivation) lives in `soundboardLibraryLogic.ts`, split out specifically so it's unit-testable without an Electron runtime — same reason `windowSize.ts`/`hotkeyLogic.ts` are split from their wiring.
- **P2P sync reuses the file-transfer WebRTC machinery completely unmodified**, as a silent background pull instead of a user-prompted push. `hooks/useSoundboardSync.ts` diffs room peers' advertised manifests (`Peer.soundboardClips`, synced space-wide like `avatarDataUrl` — including being seeded into `join` the same way, for reconnect-correctness with no separate reannounce path) against the local cache, requests what's missing (deduped to exactly one possessor per hash — never two — bounded to `SOUNDBOARD_FETCH_CONCURRENCY` (3) simultaneous links per possessor rather than file-transfer's strictly-sequential batches, since clips are tiny and sequential ICE negotiation would make a first sync take unnecessarily long), and symmetrically serves whatever a peer requests that this device actually has. Receive-side IO plugs the `soundboard-cache-begin-write`/`-write-chunk`/`-end-write`/`-abort-write` IPC channels into the **unmodified** `ReceiverIo` interface from `webrtc/transferQueue.ts`; `-end-write` re-hashes the finished file and rejects a mismatch before accepting it into the cache, so a corrupted or spoofed transfer can never be adopted under a false name.
- **Auto-sync is fully automatic**, gated only by the master `soundboardEnabled` toggle and `soundboardAutoSyncEnabled` (both Settings → Soundboard) — not per-sender trust-gated like the file-transfer trust list. Clips are tiny (~80KB) and inert; playback spam is independently guarded (concurrency cap, per-peer cooldown — `lib/soundboardTriggers.ts`), so gating sync itself would mostly just silently break the feature for untrusted peers. `lib/soundboardTrust.ts`'s `shouldAutoSyncFrom` is kept as an explicit (currently peer-independent) seam so a future per-sender model — reusing the existing file-transfer `autoAcceptUsers` list — is a one-line change, not a redesign.
- **Playback routes through the shared master bus, never the mic graph.** `lib/soundboardPlayer.ts playClip` decodes (cached in a `Map`-based LRU, capped at 24 entries via `evictOldest`) and connects a per-trigger `AudioBufferSourceNode` → per-trigger `GainNode` → `getMasterBus() ?? ctx.destination` — the identical routing `sfx.ts`'s `playSfx` already uses, so a triggered clip mixes cleanly with live voice for free and can never be picked up by the mic. `soundboard-trigger` broadcasts room-only (like chat), not space-wide (unlike the manifest) — playback should only reach peers currently sharing a voice room. **Inbound triggers are also gated by the receiver's own per-peer mute.** `usePeerVolumes` has no separate mute flag — silence is just volume ≤ 0 — so `useSoundboardPlayback` drops any trigger whose `from` matches a peer the listener has silenced (`isSenderMuted` in `lib/soundboardTriggers.ts`, checked against the same live `volumes` map), ahead of the concurrency/cooldown gates. Muting someone silences their soundboard hits too, not just their mic. Independent of that per-peer path, `soundboardMuteOthersEnabled` (Settings → Soundboard, default off) is a blanket receiver-side opt-out — when on, `useSoundboardPlayback` drops every inbound trigger regardless of per-peer volume, while the listener's own clips still play locally.
- **Spam guard is two independent layers, at two different points in the pipeline.** `SoundboardPopover.tsx` enforces a **global** (not per-clip) **1s UI cooldown** — mirrors `ReactionPopover`'s exact pattern (`cooldown` state + `setTimeout`, all tiles disabled + greyed + "Cooldown active..." title during the window) — so a user mashing any button(s) can't fire faster than one clip/second; this is the *only* call site of `triggerClip`. Independently, `lib/soundboardTriggers.ts`'s `shouldAcceptTrigger` (250ms, **per-peer**) gates the receiving end of every OTHER peer's inbound trigger — explicitly defense-in-depth against a modified client that skips its own UI cooldown, not a duplicate of it. Don't collapse these into one gate: the UI cooldown protects everyone else from *this* user spamming; the receiver-side one protects *this* user from a peer whose client doesn't self-limit.
- **Off means off, not just hidden.** The Soundboard button + its divider don't render; `useSoundboardSync`'s effects short-circuit before touching `peers` and close any in-flight P2P links rather than merely blocking new ones; the inbox watcher re-checks the **live** `soundboardEnabled` setting (not a cached value) before spawning ffmpeg, so flipping the setting off stops new spawns immediately without tearing the watcher down; disabling also re-broadcasts an **empty** manifest — a real retraction to already-connected peers, not just a stopped announcement.

### Performance guardrails

- **Animations are gated on window focus.** The app pays full animation/paint cost when **visible but unfocused** (e.g. 2nd monitor while a game has focus). `hooks/useWindowFocus.ts` tracks `windowFocused` via `window` focus/blur and App toggles `.app--unfocused` on the root; CSS under that class freezes infinite animations (e.g. the presence-dot pulse) while leaving **static** cues intact. Use `window` focus/blur (or main's `isMinimized()`), **not** `document.hidden` — it doesn't flip when the Electron window is merely minimized. New continuous animations should animate `transform`/`opacity` and be gatable under `.app--unfocused`.
- **Speaking cue is one static, accent-colored visual (no ripple).** `ParticipantTile` sets `--accent`/`--accent-glow` CSS vars from the peer's color; `.tile__avatar::after` draws a ring + glow (camera off) and `.tile--speaking` a border + glow (camera on), toggled by `showSpeaking = speaking && windowVisible` (never renders while minimized). It's a static opacity toggle — do not reintroduce a continuously-animated speaking effect.
- **Speaking detection is already lean — don't regress it.** (1) `selfSpeaking = transmitting` (i.e. the mic gate): in both modes the mic is gated (Voice = VAD via `useVoiceActivation`, PTT = key), so a live mic *is* the speaking signal — no self analyser. Remote peers run no analyser either; their state arrives off the wire. (2) The broadcast is **edge-triggered**: the `speaking-state` send is a `useEffect` keyed on the boolean, over the signaling WebSocket — not P2P media. (3) Render is a static CSS class toggle. Anti-patterns: do **not** add a per-remote-tile analyser, and do **not** make the broadcast continuous/interval-based.
- **Game detection was removed** (Windows `tasklist` scanner + `game-state` mirror) for performance — **do not reintroduce a `game-state` mirror message** expecting it to exist.

### Profile & chat sync

- **Accent color is a synced profile attribute (mirror pattern, space-wide).** `Peer.accentColor` + `PersistedSettings.accentColor` (`''` = unset). When set it's the user's accent everywhere (avatar gradient/ambient + speaking ring/glow + sidebar entry); when `''` it falls back to the auto color — `useUserColors` (grid) / `SELF_COLOR` gold for self / `userColor(userId)` hash for the sidebar. `App.tsx handleSaveAccent` sends it live; `reannounceLocalState` re-sends on reconnect.
- **Avatar sync is signaling-only, not DataChannel.** `Peer.avatarDataUrl` flows through the signaling server (sent in `join`, included in every `welcome`/`peer-joined`/`space-presence`/`space-peer-update`; live changes via `avatar-state`; re-sent by `reannounceLocalState` on reconnect). **Do not add DataChannels for avatar transfer** — the relay works space-wide (across rooms) while DataChannels are room-only. The local avatar is read from `store.getAvatarDataUrl()`/`localAvatarUrl` as the immediate ground truth; the signaling copy lags one round-trip.
- **Chat TTS (Web Speech) gotchas — `lib/tts.ts` + `lib/voices.ts`.** `speechSynthesis.speak()` self-queues, so messages serialize without a custom queue. Two Chromium quirks handled deliberately: (1) in-flight utterances are held in a module-level `Set` until `onend`/`onerror` so GC can't cut playback off; (2) the `voiceschanged` listener is attached exactly once with an idempotent refresh (voices load async). Gender matching is **heuristic** (no gender field): `voicePreference` syncs a generic **category id**, never a system voice name, + a pitch shift differentiates male/female. The picker controls **how others hear you** — gated by each *listener's* own `chatTtsEnabled`, so it stays enabled regardless of your own read-aloud toggle.

### Design system & assets

- **Ultra-Lean design system — compose primitives, use tokens, don't hardcode.** The identity is **"Blaze"**: dark = Almost Black `#070D0D` canvas + Blaze Orange `#FF6700`; light = the same signature on warm paper (`#070D0D`-family ink). Surfaces are a **4-step shade ladder** — `--bg` (stage canvas + recessed input wells) < `--panel` (shell chrome: title bar + sidebar + control bar deliberately read as ONE continuous frame) < `--card` (in-shell floats: tiles, chat panel, space-banner card, crash box) < `--float` (overlays: menus/popovers, modals, settings panel, toasts, transfer cards) — and shell regions separate by **shade only, never hairlines** (`--border` stays for form controls *inside* surfaces; shadows assist floating overlays). `theme.css` holds terse **visual-primitive** tokens (never location-named): radius `--r-1/2` (4/12px) with **enforced roles — `r-1` = interactive controls + inline elements (incl. room rows + the control deck), `r-2` = floating surfaces + media/avatars, `50%` only dots/knobs**; spacing `--s-1..6` (pure 4px); type `--fs-1..4` + weight `--fw-1/2` + line-height `--lh-1/2` (1.25 UI / 1.5 reading); `--veil-1/2` (ink hover/active overlays — the dark block redefines them stronger, 8/15% white, for the near-black canvas); `--accent-ink` (the accent as *text or focus ink* — raw `--orange` in dark, burnt orange in light where `#FF6700` fails on paper; fills + hover-borders stay raw `--orange`); `--active-bg`/`--active-text` = **solid blaze + `--on-accent` near-black ink in BOTH themes** (orange owns "active"; `--on-accent` is `#070D0D` everywhere — white on `#FF6700` is 2.9:1 and must never be used); `--glow` (the ONE live-moment halo, orange — the old violet `--secondary` is gone); media invariants `--media`/`--on-media`/`--scrim-media(-2)` for surfaces/scrims/ink over live video & photos (theme-invariant — never raw `#000`/`#fff` in components); `--sh-1/2` elevation + `--danger-fg` (label ink on danger tints — raw `--red` fails contrast there; dark `--red` is `#D6132B` so red fills clear the near-black chrome), `--dur-1/2`/`--ease` motion (`--ease` is *the* easing — every transition names explicit properties, never `transition: all`), `--danger-bg`, and an overlay `--z-dropdown/modal/nested/tooltip` scale. **No `backdrop-filter` anywhere** — the app renders live video, where backdrop blurs re-rasterize per frame; overlays are near-opaque `color-mix()` surfaces + `--sh-1` (the one deliberate element `filter: blur` — the banner name pill — is documented in place). Animate only `transform`/`opacity` (+ cheap color props); never layout props (the volume-pill/room-bar "grows" are discrete snaps with compositor fades). `styles.css` provides composable primitive classes — `.btn` (+`--primary`/`--ghost`/`--danger`/`--danger-soft`/`--block`), `.icon-btn`, `.input`, `.label`, `.backdrop` (+`--scrim`), `.menu-surface` (+`--frosted`), `.avatar`. **Avatars are rounded-square** (`.avatar` 1:2:4 scale — `--sm` 24 / base 48 / `--lg` 96px, radius `--r-1`/`--r-2`), **never `border-radius: 50%`** (that's only presence dots + knobs); avatar initials are `--on-accent` (near-black on the identity fills). **Stack a primitive + a thin residual class** for structural-only props (position/size); don't mint per-location tokens or reintroduce raw hex/px for color/radius/font-size/spacing — prefer `color-mix()` from one base. Shared menu/popover/chat entrance is `@keyframes popIn`. Genuinely-specialized buttons stay bespoke (`.ctrl-btn`, `.pill`, `.send-btn`, `.room-row__mini-btn` [derives from `--active-text` on the blaze active row], `.winctl__btn`). `var(--accent)`/`--accent-glow` in `.tile*` are runtime vars injected by `ParticipantTile`, not theme tokens; the canonical 8-color identity palette (blaze-first, `SELF_COLOR`) lives in `lib/userColors.ts` and `lib/settings.ts userColor()` hashes over the same list. Full phase-by-phase record in project memory.
- **Boost sliders restyle the native range (`-webkit-` only).** `SettingsSlider` paints a two-tone track (orange up to the boost threshold, red beyond it) via `::-webkit-slider-runnable-track` + the runtime `--fill`/`--boost` vars; this **replaces** the native `accent-color`. The room chevron-menu + per-tile sliders use the simpler whole-bar `accentColor: orange when >100%` instead.
- **Room SVG icons.** Custom SVGs load dynamically from `apps/desktop/src/renderer/src/assets/room-icons/` via Vite `import.meta.glob` (`query: '?raw'`) in `components/RoomIcon.tsx`, rendered inline with `dangerouslySetInnerHTML` and styled `fill: currentColor !important` so they match active/hover/dimmed text. `RoomModal.tsx` lists them with a search filter.
- **Branding:** the logo lives at `apps/desktop/src/renderer/src/assets/chickadee-logo.svg` (bundled by Vite); shown via `components/Logo.tsx` and rasterized to the tray by `lib/trayIcon.ts` (canvas → PNG data URL → main `Tray`).

## Constraints

Per-room cap is a unified **8** (hybrid rooms); the golden-ratio media model — one high-quality stage stream + compressed thumbnails, bounded by the per-user upload budget — is what keeps an 8-way video mesh safe. Signaling is in-memory, single-process, no auth/persistence. Windows-first (system-audio loopback); other platforms fall back to video-only screen share.
